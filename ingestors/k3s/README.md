# KepServerLogger K3S Migration

# Purpose

This runbook documents the intended execution order for migrating the KepServerLogger ingestor stack from Docker Compose to K3S/Kubernetes.

It is written for the final deployment path we selected:

- Single-node K3S server.
- Server has no internet access.
- No private registry for the first deployment.
- Images are built on a build workstation, transferred through BeyondTrust file transfer, and imported manually into K3S/containerd.
- Runtime files are managed with Kustomize-generated ConfigMaps.
- Secrets are created directly in the K3S cluster.
- TimescaleDB data is migrated from Docker to K3S using `pg_dump` and `pg_restore`.

# Architecture Summary

The migration only affects the Linux ingestor side of KepServerLogger.

Components deployed to K3S:

- TimescaleDB `StatefulSet`.
- TimescaleDB internal `Service` named `timescaledb`.
- Optional TimescaleDB external `NodePort` Service named `timescaledb-nodeport` for Grafana access from the KepServer/extractor host.
- TimescaleDB PVC named `timescaledb-data`.
- Ingestor `Deployment`.
- Ingestor certificate PVC named `ingestor-certs`.
- `kepserverlogger-env` ConfigMap for non-secret environment variables (`APPLICATION_NAME`, `DB_HOST`, `DB_PORT`).
- `kepserverlogger-files-<hash>` ConfigMap generated by Kustomize from `settings.json` and every server's tag CSV.
- `kepserverlogger-servers-<hash>` Secret generated by Kustomize from `servers.json` (per-server OPC UA URL, credentials, `db_name`, cert paths).
- `kepserverlogger-secret` Secret for TimescaleDB `DB_USER`/`DB_PASSWORD`.

Important rule:

Do not run the Docker ingestor and the K3S ingestor at the same time in normal operation. They read the same KepServer tags and can create duplicate or divergent data.

# 1. Prepare The Deployment Files On The Build Workstation

From the repository root, make sure these files exist:

```
kustomization.yaml
settings.json
servers.json
tags/<server>_tags.csv   # one CSV per entry in servers.json, path matching csv_filename
k3s/namespace.yaml
k3s/configmap.yaml
k3s/ingestor.yaml
k3s/timescaledb.yaml
k3s/timescaledb-nodeport.yaml
```

`servers.json` declares every KepServer instance the ingestor connects to (URL, credentials, `db_name`, cert paths, tag CSV). It is not committed to git — copy it from the ingestor's Docker Compose deployment, or from `docs/servers.example.json`. `kustomization.yaml`'s `configMapGenerator`/`secretGenerator` entries must list every tag CSV file declared in `servers.json`; add or remove an entry there whenever you add or remove a server.

Because the final deployment uses manual image import, the manifests must not reference `registry.local` or `imagePullSecrets`.

Run:

```bash
grep -R "registry.local\|imagePullSecrets" kustomization.yaml k3s
```

Expected result:

```
(no output)
```

Render the final image names:

```bash
kubectl kustomize . | grep "image:"
```

Expected result:

```
image: kepserverlogger/ingestors:0.1.0
image: timescale/timescaledb:latest-pg16
image: timescale/timescaledb:latest-pg16
```

TimescaleDB appears twice because it is used by the database `StatefulSet` and by the ingestor initContainer that waits for DB readiness.

# 2. Build The Ingestor Image For The Server Architecture

Build the image on a workstation that has Docker and the source code.

The K3S server architecture in this deployment is `linux/amd64`. If the build workstation architecture differs from the server, always specify `--platform`.

```bash
docker buildx build \
  --platform linux/amd64 \
  -f ingestors/Dockerfile \
  -t kepserverlogger/ingestors:0.1.0 \
  --load .
```

Save the image:

```bash
docker save kepserverlogger/ingestors:0.1.0 -o ingestors-0.1.0-amd64.tar
gzip -f ingestors-0.1.0-amd64.tar
```

Expected artifact:

```
ingestors-0.1.0-amd64.tar.gz
```

# 3. Prepare The TimescaleDB Image

TimescaleDB must also be available offline on the K3S server.

On the build workstation:

```bash
docker pull --platform linux/amd64 timescale/timescaledb:latest-pg16
docker save timescale/timescaledb:latest-pg16 -o timescaledb-latest-pg16-amd64.tar
gzip -f timescaledb-latest-pg16-amd64.tar
```

Expected artifact:

```
timescaledb-latest-pg16-amd64.tar.gz
```

If a known-good TimescaleDB tarball already exists from the previous Docker deployment, it can be reused if it imports correctly into K3S and has the expected tag.

# 4. Package Deployment Manifests And Runtime Files

From the repository root on the build workstation:

```bash
tar -czf kepserverlogger-k3s-deploy.tar.gz \
  k3s \
  kustomization.yaml \
  settings.json \
  servers.json \
  tags
```

`servers.json` contains plaintext OPC UA credentials — protect the tarball and the working directory (`chmod 600` / restrict access) on both the build workstation and the K3S server.

Expected artifact:

```
kepserverlogger-k3s-deploy.tar.gz
```

# 5. Transfer Files To The K3S Server

Use BeyondTrust file transfer to copy these files to the K3S server:

```
ingestors-0.1.0-amd64.tar.gz
timescaledb-latest-pg16-amd64.tar.gz
kepserverlogger-k3s-deploy.tar.gz
```

In the K3S Server, create a working directory:

```bash
mkdir -p /tmp/KepServerLogger
```

Place the files in this directory.

# 6. Prepare The Deployment Directory On The Server

On the K3S server:

```bash
tar -xzf /tmp/KepServerLogger/kepserverlogger-k3s-deploy.tar.gz -C /tmp/KepServerLogger
cd /tmp/KepServerLogger
```

Validate files:

```bash
ls kustomization.yaml settings.json servers.json
ls tags
ls k3s
```

Expected structure:

```
/tmp/KepServerLogger/
  kustomization.yaml
  settings.json
  servers.json
  tags/
    <server>_tags.csv   # one per servers.json entry
  k3s/
    namespace.yaml
    configmap.yaml
    ingestor.yaml
    timescaledb.yaml
    timescaledb-nodeport.yaml
```

# 7. Validate K3S Access

On the K3S server:

```bash
kubectl get nodes
```

Expected result: one Ready node.

For a quick count:

```bash
kubectl get nodes --no-headers | wc -l
```

# 8. Import Images Into K3S

Import the ingestor image:

```bash
sudo k3s ctr images import /tmp/KepServerLogger/ingestors-0.1.0-amd64.tar.gz
```

Import TimescaleDB:

```bash
sudo k3s ctr images import /tmp/KepServerLogger/timescaledb-latest-pg16-amd64.tar.gz
```

Validate ingestor image:

```bash
sudo k3s ctr images ls | grep kepserverlogger
```

Validate TimescaleDB image:

```bash
sudo k3s ctr images ls | grep timescaledb
```

Expected image tags:

```
docker.io/kepserverlogger/ingestors:0.1.0
docker.io/timescale/timescaledb:latest-pg16
```

Expected platform:

```
linux/amd64
```

# 9. Create Namespace

From `/tmp/KepServerLogger`:

```bash
kubectl apply -f k3s/namespace.yaml
```

# 10. Create Kubernetes Secrets

Create the database secret. Per-server OPC UA credentials (URL, username, password, cert paths) live in `servers.json` instead, and are turned into the `kepserverlogger-servers` Secret automatically by `kustomize`'s `secretGenerator` in step 11 — nothing to create manually for those.

```bash
kubectl create secret generic kepserverlogger-secret \
  -n kepserverlogger \
  --from-literal=DB_USER='postgres' \
  --from-literal=DB_PASSWORD='<DB_PASSWORD>'
```

Validate the secret exists:

```bash
kubectl -n kepserverlogger describe secret kepserverlogger-secret
```

This shows keys, not values.

If troubleshooting DB credential issues, inspect values carefully:

```bash
kubectl -n kepserverlogger get secret kepserverlogger-secret -o jsonpath='{.data.<SECRET_VAR>}' | base64 -d; echo
```

To inspect the OPC UA credentials from the generated `kepserverlogger-servers-<hash>` Secret:

```bash
SECRET_NAME=$(kubectl -n kepserverlogger get secret -o name | grep kepserverlogger-servers)
kubectl -n kepserverlogger get "$SECRET_NAME" -o jsonpath='{.data.servers\.json}' | base64 -d | jq .
```

# 11. Deploy K3S Resources

Apply all resources:

```bash
kubectl apply -k .
```

Immediately pause the K3S ingestor if a DB restore is still pending:

```bash
kubectl -n kepserverlogger scale deployment/ingestor --replicas=0
```

Check resources:

```bash
kubectl -n kepserverlogger get pods
kubectl -n kepserverlogger get pvc
```

Wait for TimescaleDB:

```bash
kubectl -n kepserverlogger rollout status statefulset/timescaledb
```

# 12. Stop The Docker Ingestor Before Final Backup

Do not let Docker ingestor and K3S ingestor run at the same time during cutover.

On the Docker host, from the Docker Compose directory:

```bash
cd ~/kepserverlogger/ingestors
docker compose stop ingestors
docker compose ps
```

Keep the Docker TimescaleDB container running so the backup can be created.

# 13. Create The Final Docker TimescaleDB Backup

Each entry in `servers.json` has its own `db_name` (e.g. `kepserver_pcn`, `kepserver_dmz`). Repeat this step for every `db_name` — the steps below use `<DB_NAME>` as a placeholder.

On the Docker host:

```bash
docker exec timescaledb pg_dump \
  -U postgres \
  -d <DB_NAME> \
  -Fc \
  -f /tmp/<DB_NAME>-final.dump
```

Copy the dump from the Docker container to the host:

```bash
docker cp timescaledb:/tmp/<DB_NAME>-final.dump ./<DB_NAME>-final.dump
ls -lh ./<DB_NAME>-final.dump
```

Warnings about circular foreign-key constraints on TimescaleDB internal tables can appear during `pg_dump`. These warnings are expected for TimescaleDB internals when doing a full custom-format dump and do not necessarily mean the backup failed.

# 14. Copy Backup Into The K3S TimescaleDB Pod

From the host where `<DB_NAME>-final.dump` exists, for every `<DB_NAME>`:

```bash
kubectl -n kepserverlogger cp ./<DB_NAME>-final.dump timescaledb-0:/tmp/<DB_NAME>-final.dump
```

# 15. Restore The Backup Into K3S TimescaleDB

Avoid `pg_restore --clean` with TimescaleDB hypertables. It can fail with:

```
ERROR: ONLY option not supported on hypertable operations
ERROR: cannot drop inherited constraint
```

For a full overwrite, recreate the database and restore without `--clean`.

Make sure the K3S ingestor is paused:

```bash
kubectl -n kepserverlogger scale deployment/ingestor --replicas=0
```

Recreate the database, for every `<DB_NAME>`:

```bash
kubectl -n kepserverlogger exec timescaledb-0 -- psql -U postgres -d postgres -c "DROP DATABASE IF EXISTS <DB_NAME> WITH (FORCE);"
kubectl -n kepserverlogger exec timescaledb-0 -- psql -U postgres -d postgres -c "CREATE DATABASE <DB_NAME>;"
kubectl -n kepserverlogger exec timescaledb-0 -- psql -U postgres -d <DB_NAME> -c "CREATE EXTENSION IF NOT EXISTS timescaledb;"
```

Restore, for every `<DB_NAME>`:

```bash
kubectl -n kepserverlogger exec timescaledb-0 -- pg_restore \
  -U postgres \
  -d <DB_NAME> \
  /tmp/<DB_NAME>-final.dump
```

Important: if the K3S ingestor already wrote data before this restore, those K3S-only writes can be lost unless exported and reconciled first.

# 16. Validate Restored Data

For each `<DB_NAME>`, list tables:

```bash
kubectl -n kepserverlogger exec timescaledb-0 -- psql \
  -U postgres \
  -d <DB_NAME> \
  -c "\dt"
```

Check `tags` count and timestamp range:

```bash
kubectl -n kepserverlogger exec timescaledb-0 -- psql \
  -U postgres \
  -d <DB_NAME> \
  -c "SELECT COUNT(*), MIN(server_timestamp), MAX(server_timestamp) FROM tags;"
```

# 17. Expose TimescaleDB For Grafana Access

This step is required when Grafana runs outside the K3S cluster, for example on the KepServer/extractor host, and needs to read data from the K3S TimescaleDB instance.

Do not connect Grafana with the `postgres` admin user. Create a read-only user and grant it access to every `<DB_NAME>` after the database restore is complete:

```bash
kubectl -n kepserverlogger exec -it timescaledb-0 -- psql -U postgres -d postgres
```

Inside `psql`, create the Grafana user once, then repeat the `GRANT`/`ALTER DEFAULT PRIVILEGES` block for every `<DB_NAME>` (connect to each database with `\c <DB_NAME>` first):

```sql
CREATE USER grafana_reader WITH PASSWORD '<GRAFANA_READER_PASSWORD>';

\c <DB_NAME>
GRANT CONNECT ON DATABASE <DB_NAME> TO grafana_reader;
GRANT USAGE ON SCHEMA public TO grafana_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO grafana_reader;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO grafana_reader;
```

Apply the NodePort Service:

```bash
kubectl apply -f k3s/timescaledb-nodeport.yaml
```

Expected service behavior:

```
Service name: timescaledb-nodeport
Namespace: kepserverlogger
External port: 30432
Target port: TimescaleDB/PostgreSQL 5432
```

From the KepServer/Grafana host, use the K3S node IP, not `localhost`:

```
Host: <K3S_NODE_IP>
Port: 30432
Database: <DB_NAME>   # one Grafana data source per server database
User: grafana_reader
Password: <GRAFANA_READER_PASSWORD>
TLS/SSL: disable or prefer
```

If `psql` is available from the external host, test connectivity with:

```bash
psql -h <K3S_NODE_IP> -p 30432 -U grafana_reader -d <DB_NAME>
```

Security note: this exposes TimescaleDB on the K3S node network. Keep the user read-only and restrict network access to the Grafana/KepServer host where possible.

# 18. Start The K3S Ingestor

Only after DB restore and validation:

```bash
kubectl -n kepserverlogger scale deployment/ingestor --replicas=1
kubectl -n kepserverlogger rollout status deployment/ingestor
kubectl -n kepserverlogger logs deploy/ingestor -f
```

Expected startup signs:

- `servers.json` is read and every server's tag CSV loads successfully.
- Tags are extracted for each server.
- OPC UA client setup starts for each server (one coroutine per entry in `servers.json`).
- TimescaleDB connection succeeds; each server's database is created if missing.
- Ingestor starts its main loop for all servers.

# 19. OPC UA Certificate Trust

If a specific KepServer rejects the secure channel with `BadSecurityChecksFailed`, copy that server's generated ingestor certificate and trust it in that KepServer instance. Each server declared in `servers.json` gets its own cert/key pair, generated on first boot at the `cert_path`/`key_path` it declares (relative to `/app/ingestors`, e.g. `certs/balcones_pcn_cert.pem`).

Get the pod:

```bash
kubectl -n kepserverlogger get pods -l app.kubernetes.io/name=ingestor
```

Copy the certificate for the affected server (substitute its `cert_path` from `servers.json`):

```bash
kubectl -n kepserverlogger cp <INGESTOR_POD>:/app/ingestors/certs/balcones_pcn_cert.pem ./balcones_pcn_cert.pem
```

Move the certificate to that KepServer machine and trust it in its OPC UA certificate manager.

Restart the ingestor:

```bash
kubectl -n kepserverlogger rollout restart deployment/ingestor
kubectl -n kepserverlogger logs deploy/ingestor -f
```

# 20. OPC UA User Permission Check

If the error changes to `BadUserAccessDenied`, the secure channel and certificate are working, but KepServer rejected the username/password or user permissions.

Validate the affected server's credentials from `servers.json` (see step 10 for the `kepserverlogger-servers-<hash>` Secret lookup):

```bash
SECRET_NAME=$(kubectl -n kepserverlogger get secret -o name | grep kepserverlogger-servers)
kubectl -n kepserverlogger get "$SECRET_NAME" -o jsonpath='{.data.servers\.json}' | base64 -d | jq '.[] | {name, username, password}'
```

If incorrect, fix `servers.json` on the build workstation and redeploy via `kubectl apply -k .` — `kustomize` regenerates `kepserverlogger-servers` with a new content hash, which automatically triggers a rollout of anything referencing it.

Restart the ingestor:

```bash
kubectl -n kepserverlogger rollout restart deployment/ingestor
kubectl -n kepserverlogger logs deploy/ingestor -f
```

If credentials are correct, fix permissions in KepServer:

- User exists.
- User is allowed to connect over OPC UA.
- User has read permissions on required nodes.
- Security mode and policy are allowed for that user.

# 21. Final Cutover State

After successful validation:

- K3S TimescaleDB is running.
- K3S ingestor is running.
- Grafana can read K3S TimescaleDB through `timescaledb-nodeport` using the read-only `grafana_reader` user.
- Docker ingestor remains stopped.
- Docker TimescaleDB volume is retained temporarily for rollback.
- Do not run both ingestors in normal operation.

# Known Errors And Corrections

## kubectl had no cluster context

Error:

```
current-context is not set
The connection to the server localhost:8080 was refused
```

Cause: local `kubectl` was not connected to K3S.

Correction: run commands on the K3S server or configure kubeconfig.

## TimescaleDB image import digest error

Error:

```
content digest sha256:... not found
```

Likely causes: corrupt, incomplete, or incompatible image archive.

Correction options:

```bash
gzip -t timescaledb-latest-pg16.tar.gz
gunzip -k timescaledb-latest-pg16.tar.gz
sudo k3s ctr images import timescaledb-latest-pg16.tar
```

Also use a known-good Docker deployment tarball if available.

## wrong image architecture

Error evidence:

```
docker.io/kepserverlogger/ingestors:0.1.0 linux/arm64
no match for platform in manifest
```

Cause: the image was built for `linux/arm64`, but the K3S server was `linux/amd64`.

Correction:

```bash
docker buildx build --platform linux/amd64 -f ingestors/Dockerfile -t kepserverlogger/ingestors:0.1.0 --load .
```

## pg_restore --clean failed on TimescaleDB hypertables

Error:

```
ERROR: ONLY option not supported on hypertable operations
ERROR: cannot drop inherited constraint
```

Cause: `pg_restore --clean` tries to drop constraints using `ALTER TABLE ONLY`, which conflicts with TimescaleDB hypertables/chunks.

Correction: restore into a freshly recreated database without `--clean`.

## OPC UA BadSecurityChecksFailed

Error:

```
BadSecurityChecksFailed
An error occurred verifying security
```

Cause: KepServer did not trust the K3S ingestor OPC UA client certificate.

Correction: copy the ingestor client certificate from the pod and trust it in KepServer.

## OPC UA BadUserAccessDenied

Error:

```
BadUserAccessDenied: User does not have permission to perform the requested operation
```

Cause: secure channel worked, but KepServer rejected the user or permissions.

Correction: verify the Kubernetes secret credentials and KepServer user permissions.
