# IDL K3S Deployment

# Purpose

This runbook covers converting an **already-working Docker Compose ingestor deployment** (see the repository root `README.md` for the full config model and the Docker Compose deployment guide) into a K3S deployment. It assumes you already have, on some existing deployment directory:

- A running ingestor + TimescaleDB stack via `docker-compose.yml`.
- A populated `servers.json` (one entry per KepServer instance, with real credentials and `db_name`s).
- A populated `settings.json`.
- A `tags/` directory with each server's tag CSV, matching `servers.json`'s `csv_filename` values.
- Images already buildable with `build_ingestor.sh` (see root README's Build section).

Nothing about the config *model* changes for K3S — `servers.json`/`settings.json`/tag CSVs are exactly what the ingestor already reads. Only how those files reach the container changes: Docker Compose bind-mounts them live; K3S bakes them into Kustomize-generated ConfigMaps/Secrets, so a config change requires rerunning `kubectl apply -k .`, not just editing a file in place — see [Updating Configuration](#updating-configuration).

Deployment constraints for this runbook:

- Single-node K3S server, no internet access, no private registry.
- Images are built on a build workstation, transferred through BeyondTrust file transfer, and imported manually into K3S/containerd.
- TimescaleDB data is migrated from the existing Docker deployment using `pg_dump`/`pg_restore`, once per server database.

Important rule: do not run the Docker ingestor and the K3S ingestor at the same time in normal operation. They read the same KepServer tags and can create duplicate or divergent data.

# Components Deployed To K3S

- TimescaleDB `StatefulSet`, internal `Service` (`timescaledb`), PVC (`timescaledb-data`), optional external `NodePort` Service (`timescaledb-nodeport`) for Grafana.
- Ingestor `Deployment`, certificate PVC (`ingestor-certs`, mounted read-write at `/app/ingestor/certs` — same generate-on-boot behavior as the Compose `certs/` bind mount).
- `idl-env` ConfigMap: `APPLICATION_NAME`, `DB_HOST`, `DB_PORT` (everything else `IngestorConfig` needs).
- `idl-secret` Secret: `DB_USER`, `DB_PASSWORD`.
- `idl-files-<hash>` ConfigMap, generated by Kustomize from `settings.json` and every tag CSV.
- `idl-servers-<hash>` Secret, generated by Kustomize from `servers.json`.

By default, a `Deployment` pod gets a random hostname on every restart (e.g. `ingestor-6694f76f86-8m5sz`), and the OPC UA client certificate embeds that hostname (in `app_uri` and the cert's SAN, see `lib/config.py`/`utils/certgen/src/certgen.py`) — so without a fixed hostname, *every* rollout regenerates a new, untrusted cert, breaking OPC UA connectivity on every deploy. `ingestor.yaml` pins `spec.template.spec.hostname` to the K3S server's own hostname specifically to keep the cert stable across restarts — the value itself is arbitrary as far as the app is concerned (only stability matters), but using the real server hostname makes the trusted-cert list on each KepServer meaningful to admins later. Because of that, the first boot after this migration (or after any `servers.json`/image update that recreates the pod under the new fixed hostname) still needs each server's newly-generated cert trusted once in its KepServer instance — see step 8 — but subsequent restarts should not require re-trusting.

# 1. Reuse Existing Config Files, Add The K3S Manifests

From the existing deployment directory (where `servers.json`, `settings.json`, `tags/` already live), you only need to add:

```
kustomization.yaml
k3s/namespace.yaml
k3s/configmap.yaml
k3s/ingestor.yaml
k3s/timescaledb.yaml
k3s/timescaledb-nodeport.yaml
```

`kustomization.yaml`'s `configMapGenerator` explicitly lists every tag CSV filename, so it must be kept in sync whenever a server is added to or removed from `servers.json`.

# 2. Build And Save Images

Use the same build tooling as the Compose deployment (root README's Build section):

```bash
./build_ingestor.sh --timescale ingestor/Dockerfile ingestor
```

This produces `ingestor.tar.gz` and `timescaledb.tar.gz`, tagged `ingestor:latest` and `timescale/timescaledb:latest-pg16` — matching the `image:` values already in `k3s/ingestor.yaml`/`k3s/timescaledb.yaml`, so no image renaming is needed anywhere in `kustomization.yaml`.

If the build workstation's CPU architecture differs from the K3S server's, `build_ingestor.sh` won't cross-compile — build the ingestor image manually instead:

```bash
docker buildx build --platform linux/amd64 -f ingestor/Dockerfile -t ingestor --load .
docker save ingestor | gzip > ingestor.tar.gz
```

# 3. Transfer Files To The K3S Server

Use BeyondTrust file transfer to copy to the K3S server:

```
ingestor.tar.gz
timescaledb.tar.gz
kustomization.yaml
k3s/
settings.json
servers.json
tags/
```

`servers.json` contains plaintext OPC UA credentials — restrict access (`chmod 600`) on both ends.

On the K3S server, place everything under one working directory, e.g. `/tmp/IDL`, keeping `kustomization.yaml`'s relative paths intact:

```
/tmp/IDL/
  kustomization.yaml
  settings.json
  servers.json
  tags/
    <server>_tags.csv
  k3s/
    namespace.yaml
    configmap.yaml
    ingestor.yaml
    timescaledb.yaml
    timescaledb-nodeport.yaml
```

# 4. Import Images Into K3S

```bash
sudo k3s ctr images import /tmp/IDL/ingestor.tar.gz
sudo k3s ctr images import /tmp/IDL/timescaledb.tar.gz
sudo k3s ctr images ls | grep -E "idl|ingestor|timescaledb"
```

Expect `docker.io/library/ingestor:latest` and `docker.io/timescale/timescaledb:latest-pg16`, both `linux/amd64`.

# 5. Create Namespace And DB Secret

```bash
kubectl apply -f k3s/namespace.yaml

kubectl create secret generic idl-secret \
  -n idl \
  --from-literal=DB_USER='postgres' \
  --from-literal=DB_PASSWORD='<DB_PASSWORD>'
```

Per-server OPC UA credentials in `servers.json` become the `idl-servers` Secret automatically in the next step — nothing to create manually for those.

# 6. Deploy K3S Resources

```bash
cd /tmp/IDL
kubectl apply -k .
kubectl -n idl scale deployment/ingestor --replicas=0   # pause until DB restore is done
kubectl -n idl get pods
kubectl -n idl get pvc
kubectl -n idl rollout status statefulset/timescaledb
```

# 7. Migrate TimescaleDB Data From The Docker Deployment

Repeat for every `db_name` in `servers.json` (e.g. `kepserver_pcn`, `kepserver_dmz`).

Stop the Docker ingestor first (keep Docker TimescaleDB running for the dump):

```bash
cd <docker-compose-directory>
docker compose stop ingestor
```

Dump, per `<DB_NAME>`:

```bash
docker exec timescaledb pg_dump -U postgres -d <DB_NAME> -Fc -f /tmp/<DB_NAME>-final.dump
docker cp timescaledb:/tmp/<DB_NAME>-final.dump ./<DB_NAME>-final.dump
```

Transfer each `.dump` file to the K3S server, then copy into the pod and restore. Avoid `pg_restore --clean` with TimescaleDB hypertables — it fails with `ERROR: ONLY option not supported on hypertable operations`. Recreate the database instead:

```bash
kubectl -n idl cp ./<DB_NAME>-final.dump timescaledb-0:/tmp/<DB_NAME>-final.dump

kubectl -n idl exec timescaledb-0 -- psql -U postgres -d postgres -c "DROP DATABASE IF EXISTS <DB_NAME> WITH (FORCE);"
kubectl -n idl exec timescaledb-0 -- psql -U postgres -d postgres -c "CREATE DATABASE <DB_NAME>;"
kubectl -n idl exec timescaledb-0 -- psql -U postgres -d <DB_NAME> -c "CREATE EXTENSION IF NOT EXISTS timescaledb;"
kubectl -n idl exec timescaledb-0 -- pg_restore -U postgres -d <DB_NAME> /tmp/<DB_NAME>-final.dump
```

Validate:

```bash
kubectl -n idl exec timescaledb-0 -- psql -U postgres -d <DB_NAME> -c "\dt"
kubectl -n idl exec timescaledb-0 -- psql -U postgres -d <DB_NAME> -c "SELECT COUNT(*), MIN(server_timestamp), MAX(server_timestamp) FROM tags;"
```

Important: if the K3S ingestor already wrote data before this restore, those K3S-only writes are lost unless exported and reconciled first — this is why it stays scaled to 0 until now.

# 8. Start The K3S Ingestor And Trust New Certs

```bash
kubectl -n idl scale deployment/ingestor --replicas=1
kubectl -n idl rollout status deployment/ingestor
kubectl -n idl logs deploy/ingestor -f
```

Expect one `Generated certificate: certs/<file>.pem` line per server in `servers.json` on the very first boot under the new pinned hostname (see the note above `ingestor.yaml`'s `hostname` field) — every server needs its cert trusted once, even ones already trusted for the Docker deployment or a prior K3S attempt with a different hostname. Until trusted, that server fails with `BadSecurityChecksFailed`.

For each affected server, confirm the pod's actual hostname first (it must match what `ingestor.yaml` pins), then pull the cert straight from the pod — not from any older `certs/` directory left over from a previous deployment attempt, which can coincidentally share the same hostname string in its SAN while still being a completely different certificate that KepServer won't recognize:

```bash
POD=$(kubectl -n idl get pods -l app.kubernetes.io/name=ingestor -o jsonpath='{.items[0].metadata.name}')
kubectl -n idl exec "$POD" -- hostname
kubectl -n idl cp "$POD":/app/ingestor/certs/<server>_cert.pem ./<server>_cert.pem
```

Move the certificate to that KepServer machine, trust it in its OPC UA certificate manager, then:

```bash
kubectl -n idl rollout restart deployment/ingestor
kubectl -n idl logs deploy/ingestor -f
```

If a server instead fails with `BadUserAccessDenied`, the cert is trusted but KepServer rejected the credentials — verify them:

```bash
SECRET_NAME=$(kubectl -n idl get secret -o name | grep idl-servers)
kubectl -n idl get "$SECRET_NAME" -o jsonpath='{.data.servers\.json}' | base64 -d | jq '.[] | {name, username, password}'
```

Fix `servers.json` and redeploy (see [Updating Configuration](#updating-configuration)) if wrong; otherwise check the user's permissions in KepServer.

# 9. Expose TimescaleDB For Grafana (Optional)

Required only if Grafana runs outside the cluster. This uses a `NodePort` Service, not `kubectl port-forward` — a `NodePort` is a persistent Service that stays up on its own, while `kubectl port-forward` only lasts as long as that command keeps running in a terminal somewhere. If the Grafana host can already reach the K3S node's IP (e.g. it can already ping it), no additional tunneling/forwarding is needed once the `NodePort` Service is applied.

Create a read-only user, then grant it access per `<DB_NAME>`:

```bash
kubectl -n idl exec -it timescaledb-0 -- psql -U postgres -d postgres
```

```sql
CREATE USER grafana_reader WITH PASSWORD '<GRAFANA_READER_PASSWORD>';

\c <DB_NAME>
GRANT CONNECT ON DATABASE <DB_NAME> TO grafana_reader;
GRANT USAGE ON SCHEMA public TO grafana_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO grafana_reader;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO grafana_reader;
```

```bash
kubectl apply -f k3s/timescaledb-nodeport.yaml
```

From the Grafana host, connect to `<K3S_NODE_IP>:30432` (not `localhost`) with database `<DB_NAME>`, user `grafana_reader`. One Grafana data source per server database. This exposes TimescaleDB on the node network — keep the user read-only and restrict access where possible.

# 10. Final Cutover State

- K3S TimescaleDB and ingestor running, Docker ingestor stopped.
- Docker TimescaleDB volume retained temporarily for rollback.
- Do not run both ingestor instances in normal operation.

# 11. Post-Migration Cleanup

Once both sites are confirmed ingesting successfully:

- Drop any now-unused single-server database left over from a prior single-machine K3S deployment. It may still have active connections from monitoring tools or lingering sessions, in which case a plain `DROP DATABASE` fails with `database "..." is being accessed by other users`  — use `WITH (FORCE)` (PostgreSQL 13+) to terminate those sessions automatically:
  ```bash
  kubectl -n idl exec timescaledb-0 -- psql -U postgres -d postgres -c "DROP DATABASE <OLD_DB_NAME> WITH (FORCE);"
  ```
- Remove any stale certificate files left on the K3S server from earlier deployment attempts (Docker Compose or a previous single-server K3S setup) — e.g. an old `certs/` directory, or loose `.pem` files pulled out during troubleshooting. Keep only what's actually referenced by the current `servers.json`.
- Inside the running pod, `/app/ingestor/certs` may also still contain a stale `client_cert.pem`/`client_key.pem` pair from before the multi-machine conversion — harmless (unused by the current `servers.json`), but safe to delete if you want the PVC to only reflect the current config.
- Confirm the Docker Compose ingestor for these sites is stopped everywhere it might be running, so nothing duplicates the connections now that K3S owns them.

# Updating Configuration

`servers.json`, `settings.json`, and the tag CSVs are baked into content-hashed ConfigMaps/Secrets by Kustomize — editing the file on disk alone does nothing, since the running Deployment references the specific hashed object name from the last `kubectl apply -k .`.

To change a server's credentials/URL/`db_name`, or edit `settings.json`, or rotate a tag CSV:

1. Edit the file(s) (on the build workstation, or directly in the K3S server's working directory if that's faster).
2. If a server was added or removed, add or remove its CSV entry in `kustomization.yaml`'s `configMapGenerator`.
3. Re-run, from the working directory on the K3S server:
   ```bash
   kubectl apply -k .
   ```
   Kustomize regenerates the ConfigMap/Secret with a new content hash, which changes the name the `ingestor` Deployment references and triggers an automatic rolling restart — no manual `rollout restart` needed.
4. If a new server was added, expect a new untrusted cert on next boot — repeat step 8's cert trust for it.

Code changes (`lib/`, `ingestor/src/`) need a rebuild and reimport, since there's no registry to auto-pull from:

```bash
./build_ingestor.sh ingestor/Dockerfile ingestor   # on the build workstation
```

Transfer `ingestor.tar.gz`, then on the K3S server:

```bash
sudo k3s ctr images import /tmp/IDL/ingestor.tar.gz
kubectl -n idl rollout restart deployment/ingestor
```

`imagePullPolicy: IfNotPresent` with an unchanged tag means K3S won't notice the re-imported image on its own — the explicit `rollout restart` is required.

# Known Errors And Corrections

## kubectl had no cluster context

Error:

```
current-context is not set
The connection to the server localhost:8080 was refused
```

Correction: run commands on the K3S server or configure kubeconfig.

## TimescaleDB image import digest error

Error:

```
content digest sha256:... not found
```

Likely cause: corrupt, incomplete, or incompatible image archive.

```bash
gzip -t timescaledb.tar.gz
gunzip -k timescaledb.tar.gz
sudo k3s ctr images import timescaledb.tar
```

## wrong image architecture

Error evidence:

```
docker.io/library/ingestor:latest linux/arm64
no match for platform in manifest
```

Cause: the image was built for the wrong architecture. Rebuild with `--platform linux/amd64` (see step 2).

## pg_restore --clean failed on TimescaleDB hypertables

```
ERROR: ONLY option not supported on hypertable operations
ERROR: cannot drop inherited constraint
```

Cause: `pg_restore --clean` tries to drop constraints in a way that conflicts with TimescaleDB hypertables/chunks. Correction: recreate the database and restore without `--clean` (step 7).

## OPC UA BadSecurityChecksFailed

Cause: that server's K3S-generated client certificate isn't trusted in its KepServer instance yet (step 8).

## OPC UA BadUserAccessDenied

Cause: the secure channel and cert are fine, but KepServer rejected the credentials or user permissions declared in `servers.json` for that server (step 8).

# Useful Commands

```bash
kubectl -n idl get pods
kubectl -n idl logs deploy/ingestor -f
kubectl -n idl logs deploy/ingestor --previous
kubectl -n idl describe pod <POD_NAME>
kubectl -n idl get events --sort-by=.lastTimestamp
kubectl kustomize .
```

Search for declared KepServer IPs or OPC UA URLs before transferring files off-site:

```bash
grep -RIn "opc.tcp://" . --exclude-dir=.git --exclude-dir=__pycache__ --exclude='*.tar.gz' --exclude='*.zip'
```
